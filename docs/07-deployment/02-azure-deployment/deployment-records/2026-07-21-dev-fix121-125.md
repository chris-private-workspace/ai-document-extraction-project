## 4. 部署前查證：Azure 無 override 壓制 `${knownFormats}`

本地清理測試殘留時發現 3 筆 `merge_strategy=OVERRIDE` 的 COMPANY scope 配置，其 template 不含 `${knownFormats}`，使 FIX-115 的注入對那些公司完全失效。部署前以 Kudu 唯讀查證 Azure 是否有同類配置：

```
=== Azure DEV：STAGE_2_FORMAT_IDENTIFICATION 配置共 1 筆 ===
V3.1 Stage 2 - Format Identification
  GLOBAL 啟用中 | 策略=OVERRIDE | v4 | 模板長度=255
  knownFormats: 模板=N 系統提示=Y | 提及 matchedKnownFormat=Y
```

**無任何 COMPANY / FORMAT scope 配置** → 不存在非全域 override 壓制。`v4` 即 7/20 經 Kudu 寫入的 FIX-115 + FIX-121 版本。

> ⚠️ **`模板=N` 不代表壞掉**：`${knownFormats}` 位於 `system_prompt` 而非 `user_prompt_template`，而 `stage-2-format.service.ts:164-165` 對 system 與 user **兩者都呼叫 `replaceVariables`**，變數正常展開。若僅憑此欄判讀會誤以為 Azure 的 FIX-115 同樣失效，導向相反處置。

**結論**：Azure CEVA 的 FIX-115 失效**無第二成因**，純粹源於格式散落（§5）。部署後的觀察基準是乾淨的。

## 5. 🔴 CEVA 存量：本次部署無法改善

部署前以 Kudu 唯讀盤點 Azure 存量：

| 公司 | 狀態 | 文件 | 格式 |
|------|------|------|------|
| `CEVA LOGISTICS (HONG KONG) LTD` | ACTIVE | **212** | 2（`INVOICE/OCEAN_FREIGHT` + `INVOICE/GENERAL`） |
| 其餘 6 間 CEVA 變體 | MERGED | 0 | **各 1，全部為 `INVOICE/GENERAL`** |

`DocumentFormat` 唯一鍵為 `(companyId, documentType, documentSubtype)`。存活公司已有一筆 `INVOICE/GENERAL`，故那 6 個孤立格式若執行轉移將**全部撞鍵、一個都進不去**。

**FIX-125 的程式修復對這批存量完全無效**，它只保證今後的合併正確。存量修復需先人工判定「6 個格式代表幾種真實版面」，再分配不同 `documentSubtype` 或合併刪除 —— 參考本地經驗：3 個 DHL 孤立格式經查證實為**同一版面被重複建立三次**（見 [FIX-125](../../../../claudedocs/4-changes/bug-fixes/FIX-125-company-merge-orphans-document-formats.md) 存量章節）。

另有 `7448b7c5` 名下遺留 `fieldDefinitionSets` × 1、`templateFieldMappings` × 4。

### 順帶發現：公司重複問題範圍大於預期

| 公司 | 狀態 | 文件 | 名下格式名稱 |
|------|------|------|-------------|
| `RICHON ASIA PACIFIC OPERATIONS LIMITED（…）` | ACTIVE | 1 | 「CEVA Logistics - Original Invoice（紙本/掃描版）」 |
| `RICON ASIA PACIFIC OPERATIONS LIMITED` | ACTIVE | 3 | 「CEVA Logistics Original Invoice（多欄位運輸/清關型）」 |

兩間 ACTIVE 公司名下掛著明顯屬於 CEVA 的格式 → CEVA 文件被識別到別的公司。另 `Nippon Express` 有三間並存（53 / 16 / 1 份文件）。屬 CHANGE-103 治理範圍，本次未處理。
