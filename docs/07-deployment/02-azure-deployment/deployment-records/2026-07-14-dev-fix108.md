## 4. 🔴 事故：整批 `OCR_FAILED`（部署後 20 分鐘內）

### 症狀
部署後上傳的 10 份文件全滅：7 份 `OCR_FAILED`、3 份 `REF_MATCH_FAILED`（後者為既有 ref match 設計行為，無關）。部署前 04:12 同來源的 10 份全部 `MAPPING_COMPLETED`。

### 根因
CHANGE-100/102 改用新 env 解析 Azure OpenAI 部署名，但 **Azure App Service 從未設定這兩個新 env**：

| 程式碼要讀的 env | Azure 現況 | 靜默 fallback 值 | Azure 實際部署名 |
|---|---|---|---|
| `AZURE_OPENAI_GPT54_MINI_DEPLOYMENT_NAME` | ❌ 不存在 | `gpt-5.4-mini` | `gpt-5.4-mini-aidocprocessing` |
| `AZURE_OPENAI_GPT54_NANO_DEPLOYMENT_NAME` | ❌ 不存在 | `gpt-5.4-nano` | `gpt-5.4-nano-aidocprocessing` |

→ Stage 1 呼叫不存在的部署 → `404 DeploymentNotFound` → `OCR_FAILED`。
（次生災情：每份重試 5 次 × 並發 → 1 份另報 `Transaction API error: Unable to start a transaction`。）

### 解法（未回滾映像）
```bash
az webapp config appsettings set -g RG-RAPOSCM-AIDocProcessing-DEV -n WebApp-RAPOSCM-AIDocProcessing-DEV \
  --settings AZURE_OPENAI_GPT54_MINI_DEPLOYMENT_NAME=gpt-5.4-mini-aidocprocessing \
             AZURE_OPENAI_GPT54_NANO_DEPLOYMENT_NAME=gpt-5.4-nano-aidocprocessing
```
→ 自動重啟 → 新容器 uptime 歸零、health 200 + database connected。**失敗的 10 份文件需重新處理。**

### 預防（已落實）
- runbook **§A.0** 新增前置檢查：部署前必查「線上映像 commit → 目標 commit」的 `.env.example` diff，逐一對照 Azure app settings 補齊。
- runbook **§16** 完整記錄此事故。
- ⚠️ `.env.example` 本來就列了這兩個 env，註解甚至預告了 `DeploymentNotFound` —— 地雷有標示，是部署前沒去 diff 它。
