## 一、執行摘要（Executive Summary）

### 1.1 目前狀況

Reference Number 批次匯入功能目前存在 **2 個已確認缺陷** 與 **4 個連帶風險**。使用者以 5000 筆的 Excel 匯入時，實際只有 1000 筆進入系統，且即使填寫了「文件子類型（Document Sub Type）」也全部變成空值。經完整程式碼稽核，這些並非單一 bug，而是反映了此功能在「小量資料」假設下設計、未考慮規模化的結構性問題。

### 1.2 本方案定位

本方案聚焦於 **不改動核心架構、風險低、可立即實施** 的修正，目標是：

1. **立即止血** — 修復資料遺漏（Sub Type）與靜默截斷（1000 筆上限）兩個直接影響資料正確性的缺陷。
2. **消除誤導** — 讓前端顯示的「驗證通過」與後端實際匯入結果一致，避免使用者誤以為匯入成功。
3. **有限度提升容量** — 在瀏覽器解析能力可承受的範圍內，將單檔上限由 1000 筆提升至 5000 筆。
4. **強化可靠性** — 加入分批交易與更清楚的錯誤回報，降低「部分成功、無法追蹤」的風險。

> **本方案無法解決 10 萬筆等級的大量匯入需求。** 該需求牽涉架構層級的重新設計，請見長期方案（方案二）。本方案是長期方案的「Phase 0」，兩者互補而非互斥。

### 1.3 關鍵數據

| 指標 | 現況 | 本方案完成後 |
|------|------|--------------|
| 單檔匯入上限 | 1000 筆（靜默截斷，無警示） | 5000 筆（超出時明確提示） |
| Document Sub Type 匯入 | 100% 遺失（新增路徑漏寫） | 正確寫入 |
| 前端「驗證通過」可信度 | 低（不檢查地區是否存在等） | 高（與後端規則一致） |
| 匯入中途失敗的資料一致性 | 無交易保護，部分寫入 | 分批交易，批次內全有或全無 |
| 預估總工作量 | — | **約 5–8 人天** |

## 二、問題背景與影響

### 2.1 問題全景

經端到端稽核（前端解析 → API → Zod 驗證 → 服務層 → 資料庫），發現的問題分為三類：

| 編號 | 問題 | 類別 | 業務影響 | 嚴重性 |
|------|------|------|----------|--------|
| P-1 | 單檔僅匯入前 1000 筆，超出部分被靜默丟棄 | 缺陷 | 資料大量遺漏且無警示，使用者以為全部匯入 | 🔴 高 |
| P-2 | Document Sub Type 於新增時未寫入資料庫 | 缺陷 | 匯入資料缺欄位，影響後續進出口方向判斷 | 🔴 高 |
| P-3 | 前端「驗證通過」與後端實際匯入結果不一致 | 缺陷 | 誤導使用者，綠燈卻未匯入 | 🟠 中高 |
| P-4 | 同一檔案內重複資料互相跳過/覆蓋，無明確提示 | 風險 | 匯入筆數少於預期，難以排查 | 🟡 中 |
| P-5 | 整批匯入無交易保護，中途失敗留下半套資料 | 風險 | 資料一致性受損，重試產生重複 | 🟡 中 |
| P-6 | 流水號（code）以隨機碼生成，大量時碰撞失敗 | 風險 | 大量匯入時零星筆數失敗 | 🟡 中 |

### 2.2 業務影響說明（供管理層參考）

- **資料可信度風險**：P-1、P-2 直接造成「使用者相信已匯入，實際資料不完整」。若這些 Reference Number 被下游的發票比對流程使用，會導致比對失敗或錯誤，且問題難以回溯。
- **人力浪費**：使用者需反覆嘗試、手動核對哪些資料進了系統，抵銷了批次匯入原本要節省的人力。
- **信任損耗**：功能「看起來成功、實際失敗」是最傷使用者信任的失效模式。

## 三、改善範圍與原則

### 3.1 範圍界定

| 納入本方案（中短期） | 不納入（屬長期方案） |
|----------------------|----------------------|
| 修復 Sub Type 漏寫 | 支援 10 萬筆等級匯入 |
| 前後端驗證一致性 | 後端串流解析 Excel |
| 上限 1000 → 5000 | 非同步背景任務處理 |
| 分批交易 + 錯誤回報 | 資料模型層級變更（ImportJob） |
| 流水號碰撞緩解 | 上傳流程 UX 全面重構 |

### 3.2 設計原則

1. **不動核心架構** — 維持「前端 ExcelJS 解析 → 單次 POST → 服務層迴圈寫入」的既有骨架，只在其中做外科手術式修正。
2. **向後相容** — 不改變 API 契約的既有欄位、不改變資料庫既有欄位語意；符合既有 Zod / RFC 7807 規範。
3. **可獨立交付** — 每個改善項目可獨立完成、獨立驗證、獨立上線，降低風險。
4. **符合專案規範** — 遵循 CLAUDE.md 之 i18n 同步、type-check、lint 等完成前檢查。

## 四、改善項目明細

> 以下每個項目均標註：問題根因、解決方案、涉及檔案、工作量估算、風險、驗收標準。工作量為初步估算，需開發團隊細化。

### SI-1　修復 Document Sub Type 未寫入（對應 P-2）

**根因**：`reference-number.service.ts` 的 `importReferenceNumbers` 函數有「新增」與「更新」兩條路徑。更新路徑（overwriteExisting）有寫入 `documentSubType`，但**新增路徑的 `prisma.referenceNumber.create` 資料物件遺漏了此欄位**。批次匯入絕大多數為新增，故 Sub Type 全數遺失。

**解決方案**：於新增路徑的 `create` 資料物件補上 `documentSubType: item.documentSubType ?? null`，與更新路徑及 `createReferenceNumber`（單筆建立，本身正確）對齊。

**涉及檔案**：`src/services/reference-number.service.ts`（新增路徑，1 行）

**工作量**：0.25 人天（含測試）

**風險**：極低。單一欄位補寫，不影響其他邏輯。

**驗收標準**：
- 匯入含 Sub Type 的 Excel，新增記錄的 `documentSubType` 正確寫入資料庫。
- Sub Type 為空或非法值時，寫入 NULL（不報錯）。
- 迴歸測試：更新路徑（overwriteExisting）行為不變。

---

### SI-2　前後端驗證一致性（對應 P-3）

**根因**：前端 `validateRow` 只檢查 4 項（number 長度、type 合法、year 範圍、regionCode 非空），但後端額外要求：
- `regionCode` 必須對應資料庫**實際存在**的地區（服務層查無即該筆失敗）。
- `regionCode` 長度 2–20、`description` ≤500、`code` ≤50（Zod 規則）。

前端顯示綠色「驗證通過」，但這些後端規則未在前端檢查，導致「綠燈卻未匯入」。

**解決方案**：
1. 前端在解析階段載入現有地區清單（可複用既有 region API / hook），對每筆 `regionCode` 檢查是否存在，不存在則標為無效並提示。
2. 前端 `validateRow` 補齊 `regionCode`（2–20）、`description`（≤500）、`code`（≤50）長度檢查，與後端 Zod 一致。
3. 驗證訊息 i18n 化（en / zh-TW / zh-CN 同步），不硬編碼。

**涉及檔案**：
- `src/components/features/reference-number/ReferenceNumberImportDialog.tsx`（驗證邏輯 + 地區比對）
- `messages/{en,zh-TW,zh-CN}/referenceNumber.json`（新增驗證訊息）

**工作量**：1.5 人天

**風險**：低。純前端驗證增強，不影響後端。需注意 i18n 三語言同步（H5）。

**驗收標準**：
- 填入不存在的 `regionCode`，前端即顯示無效並說明原因，該筆不計入「valid」。
- 超長欄位在前端即被標為無效。
- 前端顯示的 valid 筆數 = 後端實際成功匯入筆數（在無重複的前提下）。
- `npm run i18n:check` 通過。

---

### SI-3　有限度調高單檔上限（對應 P-1）

**根因**：上限 1000 寫死於兩處 —— 前端 `MAX_ITEMS`（解析時截斷）與後端 `IMPORT_MAX_ITEMS`（Zod `.max()` 拒絕）。前端截斷是「靜默」的，使用者無從得知資料被丟棄。

**解決方案**：
1. 將兩處常數由 1000 調整為 **5000**，兩處必須同步。
2. **關鍵**：前端解析時，若實際資料列數超過上限，**不再靜默截斷**，而是明確提示「檔案含 N 筆，超過單檔上限 5000 筆，請拆分後匯入」，並阻擋提交。
3. 提示訊息 i18n 化。

> **為何是 5000 而非更高**：上限受限於「瀏覽器用 ExcelJS 解析並將全部資料存入 React 狀態」的能力。5000 筆為經驗上瀏覽器仍可流暢處理的保守值。突破此量級須改為後端解析（長期方案）。此數字可由團隊以實測微調。

**涉及檔案**：
- `src/components/features/reference-number/ReferenceNumberImportDialog.tsx`（`MAX_ITEMS` + 超限提示）
- `src/lib/validations/reference-number.schema.ts`（`IMPORT_MAX_ITEMS`）
- `messages/{en,zh-TW,zh-CN}/referenceNumber.json`（超限提示訊息）

**工作量**：1 人天（含瀏覽器實測不同筆數的效能表現）

**風險**：中低。需實測 5000 筆時的瀏覽器解析時間與記憶體，確認 UX 可接受；若卡頓明顯則下修上限。

**驗收標準**：
- 上傳 5000 筆檔案可正常解析、預覽、匯入。
- 上傳超過 5000 筆的檔案，顯示明確超限提示並阻擋提交，**不再靜默丟棄**。
- 前後端上限一致，後端不會回 400（因前端已阻擋）。

---

### SI-4　批次內重複偵測與錯誤回報（對應 P-4）

**根因**：服務層對每筆先查 `code`、再查 `number+type+year+regionId` 是否存在。同一檔案內若有重複鍵，第一筆寫入後，第二筆會被判定為「已存在」而跳過或覆蓋，使用者無明確提示為何筆數短少。此外，目前 skip 的原因回報較簡略。

**解決方案**：
1. 前端解析後，在預覽階段偵測檔案內重複鍵（相同 `number+type+year+regionCode`），以警示標示並統計。
2. 強化匯入結果回報：明確區分「新增 / 更新 / 跳過（含原因：重複、地區不存在、驗證失敗）/ 失敗」，並允許使用者下載結果明細。
3. 結果訊息 i18n 化。

**涉及檔案**：
- `src/components/features/reference-number/ReferenceNumberImportDialog.tsx`（重複偵測 + 結果明細呈現）
- `src/services/reference-number.service.ts`（回報結構補上 skip 原因分類，屬相容擴充）
- `messages/{en,zh-TW,zh-CN}/referenceNumber.json`

**工作量**：1.5 人天

**風險**：低。回報結構為向後相容擴充（新增欄位）。

**驗收標準**：
- 檔案含重複鍵時，預覽階段即提示重複筆數。
- 匯入後結果清楚顯示各類別筆數與原因，使用者可下載明細。

---

### SI-5　分批交易與可靠性（對應 P-5）

**根因**：服務層以單一 for 迴圈逐筆寫入，**未包裹任何交易**。匯入途中若逾時或連線中斷，已寫入的資料留在庫中、後續未寫入，狀態不一致；重試會產生重複或非預期更新。

**解決方案**（不改架構的折衷）：
1. 將寫入迴圈改為**分批交易**：每 N 筆（如 500 筆）包在一個 `prisma.$transaction` 內，批次內全有或全無。
2. 記錄已完成批次，逾時或失敗時回報「已成功 X 批、失敗於第 Y 批」，使用者可從失敗點續傳或重試。
3. 維持既有 `skipInvalid` 語意。

> **注意**：這是「分批交易」而非「單一大交易」。對 5000 筆而言，單一大交易會長時間鎖定且易逾時；分批交易在可靠性與效能間取得平衡。真正的大量匯入仍須長期方案的非同步處理。

**涉及檔案**：`src/services/reference-number.service.ts`（迴圈改分批交易）

**工作量**：1.5 人天

**風險**：中。需仔細處理批次邊界、既有重複偵測邏輯與交易的互動；需完整測試逾時與部分失敗情境。

**驗收標準**：
- 匯入 5000 筆，中途模擬失敗，已提交批次資料完整、未提交批次無任何殘留。
- 結果回報可指出失敗批次，支援安全重試。

---

### SI-6　流水號碰撞緩解（對應 P-6）

**根因**：`generateCode` 以 `REF-{year}-{region}-{4位隨機碼}` 生成，隨機碼僅約 168 萬種組合，最多重試 10 次；同一 year+region 資料量大時碰撞率升高，重試耗盡即該筆失敗。且每次生成都額外查詢一次資料庫。

**解決方案**（中短期輕量版）：
1. 將隨機碼位數增加（如 4 位 → 6 位，組合數提升約 1300 倍），大幅降低碰撞率。
2. 於單次匯入開始時，一次撈出該批涉及的既有 code 至記憶體集合，生成時先在記憶體檢查，減少逐筆資料庫查詢。

> **注意**：徹底解決 code 生成的效能與唯一性，應改為確定性序號策略（長期方案）。本項目為緩解，非根治。

**涉及檔案**：`src/services/reference-number.service.ts`（`generateCode` 及其呼叫處）

**工作量**：1 人天

**風險**：低。

**驗收標準**：
- 於同一 year+region 匯入 5000 筆無 code 資料，無「無法生成唯一 code」錯誤。
- 生成階段的資料庫查詢次數顯著下降。

## 七、成本效益分析

### 7.1 成本

- **開發**：約 5–8 人天（單一開發者約 1.5–2 週，含測試）。
- **不需**：新增第三方依賴、變更資料庫結構、變更部署架構、額外基礎設施費用。

### 7.2 效益

| 效益 | 說明 |
|------|------|
| 資料正確性 | 消除 Sub Type 遺失與靜默截斷兩大資料完整性缺陷 |
| 使用者信任 | 「所見即所得」— 綠燈即代表會匯入 |
| 容量 5 倍 | 單檔上限 1000 → 5000，涵蓋更多日常情境 |
| 可靠性 | 分批交易 + 明細回報，失敗可追蹤、可重試 |
| 低風險快速見效 | 不動架構，1.5–2 週內可分批上線 |

### 7.3 侷限（誠實揭露）

本方案**不解決** 10 萬筆等級的匯入需求。當單次需求突破 5000 筆、或需要背景非同步處理與即時進度時，必須進入長期方案。本方案投入的修正（尤其 SI-1、SI-2、SI-4 的驗證與回報邏輯）在長期方案中大部分可延續複用，非重工。

## 八、與長期方案的銜接

| 中短期項目 | 於長期方案的角色 |
|------------|------------------|
| SI-1 Sub Type 修復 | 直接延續（服務層寫入正確性是基礎） |
| SI-2 驗證一致性 | 演進為後端統一驗證管線的一部分 |

### 與原規劃的兩處偏離（H6 已先向用戶報告）

1. **SI-6 前提修正**：稽核發現 `generateRandomCode` 實際已產生 **6 字元**（非規劃所述「4 位」），碰撞空間約 21 億（36^6），風險本就低。故 SI-6 重點改為 **消除 `generateCode` 逐筆 `findUnique` 的 N+1 查詢**（一次預載既有 code 至記憶體 Set），並順帶將隨機碼改為固定 6 字元、避免原 base64 過濾後長度不足。

2. **SI-5 交易模式分流（正確性修正）**：原規劃「分批交易 + 維持 skipInvalid 語意」在 `skipInvalid=true`（前端預設）下有隱患——PostgreSQL 交易內任一 query 失敗會使整個交易進入 aborted，後續操作全部失敗，導致「容忍單筆錯誤」無法實現、反而連累整批。故改為依模式分流：
   - `skipInvalid=true`（容錯）：逐筆獨立寫入、不包交易，單筆錯誤只影響該筆。
   - `skipInvalid=false`（嚴格）：分批交易（每批 200 筆），任一筆失敗即該批 rollback 並中止，已提交批次保留可續傳。
